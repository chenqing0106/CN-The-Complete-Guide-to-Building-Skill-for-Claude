[https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf?hsLang=en…](https://t.co/1ZA5FtpQCj)

## 核心内容总结

**Skill 是什么**：一个文件夹，核心是 `SKILL.md`，告诉 Claude 什么时候用、怎么用某个工作流。类似"给 Agent 装插件"，但比插件更轻量——本质是结构化的 Prompt + 工具调用指南。

**三层渐进式设计**是最聪明的地方：YAML frontmatter 常驻系统提示词（决定"要不要触发"），SKILL.md 正文按需加载（决定"怎么做"），references/ 文件夹按需读取（决定"查什么资料"）。这直接解决了 Token 浪费问题。

**五种核心模式**：顺序工作流、多 MCP 协调、迭代精炼、上下文感知工具选择、领域专业知识嵌入。覆盖了绝大多数实际场景。

---

## 评价

**优点**

写得非常务实，没有学术腔。尤其两个地方特别好：一是"先在单个任务上迭代直到成功，再提取成 Skill"的建议，这是真正踩过坑得出的结论。二是 Troubleshooting 章节，把"触发太少、触发太多、指令不被遵守"这三类最常见问题都给了具体的诊断路径，不是泛泛而谈。

**局限性**

有两点值得警惕：

第一，**评估体系还很粗糙**。文档自己也承认"接受 vibes-based assessment"，量化指标（90% 触发率、0 次 API 失败）只是目标，没有提供真正的测量工具。对于认真做 Agent 的人来说，这块还需要自己补。

第二，**MCP 依赖还比较重**。很多模式都预设了你已经有 MCP Server，对于刚入门的开发者，前置成本不低。

## 引言

技能是一组打包为简单文件夹的指令，用于教导 Claude 如何处理特定任务或工作流。这是根据特定需求定制 Claude 的有效方式之一。与其在每次对话中重复说明偏好、流程和领域专业知识，技能可以让你只需教导 Claude 一次，即可持续受益。

当拥有可重复的工作流时，技能非常强大：例如生成前端设计、执行标准化的研究、创建遵循团队风格指南的文档，或协调多步流程。它们与 Claude 的内置功能（如代码执行和文档创建）配合良好。对于构建 MCP（模型上下文协议）集成的人来说，技能增加了一个强大的层，有助于将原始工具访问转化为可靠、优化的工作流。

**你将学到什么：**

- 技能结构的技术要求和最佳实践
- 独立技能和 MCP 增强工作流的模式
- 在不同用例中行之有效的模式
- 如何测试、迭代和分发你的技能

**适用人群：**

- 希望 Claude 始终如一地遵循特定工作流的开发者和高级用户
- 希望在整个组织内标准化 Claude 工作方式的团队

使用 `skill-creator` 构建并测试第一个工作技能大约需要 15–30 分钟。

## 第一章：基础知识

### 什么是技能？

技能是一个包含以下内容的文件夹：

- `SKILL.md`（要求存在）：带有 YAML 前置元数据的 Markdown 指令
- `scripts/`（可选）：可执行代码（如 Python、Bash 等）
- `references/`（可选）：根据需要加载的文档
- `assets/`（可选）：输出中使用的模板、字体、图标等

### 核心设计原则

1. **渐进式披露：** 技能使用三层系统来最小化 token 消耗：
   - 第一层（YAML 前置元数据）：始终加载到 Claude 的系统提示词中，提供触发条件判断信息。
   - 第二层（`SKILL.md` 主体）：当 Claude 认为该技能相关时加载，包含完整指令。
   - 第三层（链接文件）：按需导航和发现的附加捆绑文件。

2. **可组合性：** Claude 可以同时加载多个技能。你的技能应与其他技能协同工作。

3. **可移植性：** 技能在 Claude.ai、Claude Code 和 API 上通用，一次创建即可跨平台运行。

### 面向 MCP 构建者：技能 + 连接器

如果 MCP 提供了“专业厨房”（访问工具、数据和设备），技能则提供了“食谱”（如何创造价值的逐步说明）。

| MCP（连接性） | 技能（知识） |
|---|---|
| 将 Claude 连接到你的服务（Notion, Asana 等） | 教导 Claude 如何有效使用你的服务 |
| 提供实时数据访问和工具调用 | 捕获工作流和最佳实践 |
| 决定 Claude 能做什么 | 决定 Claude 应该怎么做 |

没有技能引导，用户连接 MCP 后可能不知道如何操作，导致支持工单增加且每次对话都要从头开始。加入技能后，预建工作流会自动激活，实现一致且可靠的工具调用。

## 第二章：规划与设计

### 从用例开始

在编写代码之前，应先确定 2–3 个具体用例。例如，项目冲刺规划：

```
用例：项目冲刺规划
触发：用户说“帮我计划这个冲刺”或“创建冲刺任务”
步骤：
1. 通过 MCP 从 Linear 获取当前项目状态
2. 分析团队速度和容量
3. 建议任务优先级
4. 在 Linear 中创建带有标签和工时估算的任务
结果：冲刺规划完成，任务生成
```

项目冲刺规划示例：

- **触发：** 用户说“帮我计划这个冲刺”或“创建冲刺任务”
- **步骤：**
  1. 通过 MCP 从 Linear 获取状态
  2. 分析团队容量
  3. 建议任务优先级
  4. 在 Linear 中创建带标签的任务

**自我提问：**
- 用户想完成什么？
- 这需要哪些多步工作流？
- 需要哪些工具（内置或 MCP）？
- 应该嵌入哪些领域知识或最佳实践？

### 常见的用例类别

- **文档与资产创建**：用于生成一致、高质量的输出，如文档、演示、应用、设计、代码等。真实示例：frontend-design 技能，其描述为“创建具有高设计质量的前端界面，用于 web 组件、页面、文档、海报或应用”。关键技术包括嵌入样式指南、模板结构、质量检查列表等，不需要外部工具，完全使用 Claude 内置能力。

- **工作流自动化**：用于一系列有条理的流程，常跨多个 MCP 服务器协同。真实示例：skill-creator 技能，“交互式创建新技能的向导，引导用户定义用例、生成前置元数据、编写指令和验证”。关键技术包括逐步流程和验证门、通用结构模板、内置审查和迭代循环。

- **MCP 增强型**：提供工作流指导以增强 MCP 服务器的工具访问。真实示例：sentry-code-review 技能（由 Sentry 提供），“使用 Sentry 的错误监控数据自动分析并修复 GitHub Pull Request 中检测到的错误”。关键技术包括串联多个 MCP 调用、嵌入领域专业知识、提供用户不必指定的上下文、处理常见 MCP 错误。

### 如何判断技能是否有效？

以下为参考性的定量和定性目标：

**定量指标**
- 在 90% 的相关查询中触发：运行 10–20 个应触发的测试查询，统计自动加载次数与显式调用次数。
- 在 X 次工具调用内完成工作流：对比启用与禁用技能的消耗，计算工具调用数和 token 总量。
- 每个工作流零失败 API 调用：在测试时监控 MCP 服务器日志，追踪重试率和错误码。

**定性指标**
- 用户不需要提示 Claude 下一步：测试时记录重定向或澄清的频率，并向内部用户征求反馈。
- 工作流无需用户纠正：对同一请求运行 3–5 次，检查输出结构和质量是否一致。
- 跨会话结果一致：新用户首次尝试是否能在最少指导下完成任务？

### 技术要求

#### **文件结构示例：**

此处用图形树
```
your-skill-name/
  SKILL.md               # 必需：主技能文件
  scripts/               # 可选：可执行代码
    process_data.py      # 示例
    validate.sh          # 示例
  references/            # 可选：文档
    api-guide.md         # 示例
  assets/                # 可选：模板等
    report-template.md   # 示例
```

#### **关键命名规则：**

- 必须精确命名为 `SKILL.md`（区分大小写）。
- 技能文件夹使用短横线命名法（如 `notion-project-setup`），不能有空格、下划线或大写字母。
- 不要包含通用的 `README.md`。

**YAML frontmatter前置元数据：**

这是 Claude 决定是否加载技能的关键。

```yaml
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
metadata:
  author: ProjectHub
  version: 1.0.0 
  mcp-server: projecthub
```

- **name（必需）：** 短横线命名法，与文件夹名称一致。
- **description（必需）：** 包含它做什么和何时使用它（触发条件）；少于 1024 字符。
- **许可（可选）：** 如果技能开源，可添加许可证说明（如 MIT、Apache‑2.0 等）。字符数 1‑500，用于指示环境要求，例如目标产品、所需系统包、网络访问等。
- **metadata（可选）：** 任意自定义键值对。建议包含作者、版本和 mcp-server，例如：
  ```yaml
  metadata:
    author: ProjectHub
    version: 1.0.0
    mcp-server: projecthub
  ```
- **安全限制：** 前置元数据中禁止出现 XML 尖括号（`< >`），名称中不能包含保留字（`claude` 或 `anthropic`），以防指令注入。

#### **编写有效技能：**

描述字段应包含 `[它做什么] + [何时使用它] + [关键能力]`。

```yaml
# 好的描述示例（包含触发短语）
description: Manages Linear project workflows including sprint planning, task creation, and status tracking. Use when user mentions "sprint", "Linear tasks", "project planning", or asks to "create tickets".


# 差的描述示例（太模糊）
description: Helps with projects.
```

**主指令推荐结构：**

```markdown
name: your-skill
description: [...]

# Your Skill Name
## Instructions
### Step 1: [First Major Step]
Clear explanation of what happens.
Example:

```bash
python scripts/fetch_data.py --project-id PROJECT_ID
```

### 示例与故障排查

在技能文档中列出典型场景、操作步骤和预期结果，并为常见错误提供排查信息。例如：

- **示例 1：创建营销活动**
  - 用户说：“设置一个新的营销活动”
  - 执行动作：
    1. 通过 MCP 获取现有活动
    2. 使用提供的参数创建新活动
  - 预期结果：活动创建成功并返回确认链接
  - 常见错误：
    - 错误：“参数缺失”
      - 原因：输入数据不完整
      - 解决：提示用户补全字段并重试

### 指令编写最佳实践

- 引用捆绑资源，例如 `scripts/validate.py`。
- 内容要具体且可操作：
  - ✅ 推荐：运行 `python scripts/validate.py --input {filename}` 检查数据格式。如果验证失败，可能是缺少字段或日期格式不对（使用 YYYY‑MM‑DD）。
  - ❌ 不推荐：仅写“验证数据”，这太笼统。
- 提供错误处理和常见问题的解决步骤。

### 常见问题示例

- **MCP 连接失败**
  1. 检查 MCP 服务器是否运行：转到 设置 > 扩展。
  2. 确认 API key 有效。
  3. 尝试重新连接：设置 > 扩展 > [你的服务]。

在编写查询前，可参阅 `references/api-patterns.md` 获取：
- 速率限制指南
- 分页模式
- 错误代码及处理

### 渐进式披露

保持 `SKILL.md` 聚焦核心指令，将详细文档移至 `references/` 并链接。这符合前述的三层设计原则。

## 第三章：测试与迭代

1. 通过 Claude.ai 进行快速手动测试
2. 通过 Claude Code 进行自动化脚本验
3. 通过技能 API 进行程序化评估。

建议先针对单个具有挑战性的任务进行迭代，成功后再扩展测试用例覆盖范围。

#### **推荐的测试方法包括：**

1. **触发测试：** 确保技能在相关词汇下加载。
2. **功能测试：** 验证输出正确性、API 调用及错误处理。
3. **性能对比：** 证明技能减少了 token 消耗和错误率。

触发测试 目标：确保技能在正确的时机加载。
测试用例：
- ✅ 在明确的任务上触发
- ✅ 在同义表达下也能触发
- ❌ 不在无关话题上触发

示例测试套件：
应触发：
- “帮我设置一个新的 ProjectHub 工作区”
- “我需要在 ProjectHub 创建一个项目”
- “为第四季度规划初始化一个 ProjectHub 项目”
不应触发：
- “旧金山的天气怎么样？”
- “帮我写 Python 代码”
- “创建一个电子表格”（除非 ProjectHub 技能支持表格）

功能测试 目标：验证技能生成正确输出。
测试项：
- 输出有效
- API 调用成功
- 错误处理到位
- 覆盖边缘情况

示例：测试创建 5 个任务的项目
给定：项目名称 “Q4 Planning”，5 个任务描述
当：技能执行工作流
则：
- 项目在 ProjectHub 中创建
- 5 个任务创建并带正确属性
- 所有任务关联到项目
- 无 API 错误

性能对比 目标：证明技能相比基线有改进。
使用“定义成功标准”中的指标。一个比较示例：

基线情况（无技能）：
- 用户每次都需提供指令
- 往返对话 15 次
- 3 次 API 调用失败并需重试
- 消耗 12,000 个 token

启用技能后：
- 自动执行工作流
- 仅问 2 次澄清问题
- 0 次 API 失败
- 消耗 6,000 个 token

#### 使用 skill-creator 技能

你可以使用官方的 `skill-creator` 技能来帮助生成和审查技能草稿。当遇到触发不足、执行失败或过度触发的情况，应及时调整描述字段的关键词并添加负向触发器。

skill-creator 可在 Claude.ai 插件目录中获得，或在 Claude Code 中下载使用。只要你有 MCP 服务器并知道前 2–3 个工作流，就能在一次会话中构建并测试一个功能性技能，通常仅需 15–30 分钟。

创建技能时，它可以：
- 根据自然语言描述生成技能
- 产出格式正确的 SKILL.md 与前置元数据
- 建议触发短语和结构

审查技能时，它可以：
- 标出常见问题（描述模糊、缺少触发词、结构不当）
- 识别过度或触发不足的风险
- 根据技能目的建议测试用例

迭代改进：
- 在使用技能过程中遇到边缘情况或失败，将这些示例反馈给 skill-creator
- 例如：“使用此对话中识别的问题和解决方案，改进技能处理[特定边缘情况]的方式”

使用提示：在对话中输入“使用 skill-creator 技能帮我构建一个适用于[你的用例]的技能”。

注意：skill-creator 仅帮助设计和细化技能，不会执行自动测试套件或生成定量评估结果。

#### 基于反馈的迭代

技能是一个活文档，应根据反馈持续迭代。

- **触发不足信号**：
  - 技能未在应触发时加载
  - 用户手动启用
  - 有关何时使用的支持问题
  - 解决：在描述中添加更多细节和关键词，尤其是技术术语

- **触发过多信号**：
  - 技能在无关查询中加载
  - 用户禁用技能
  - 目的不清
  - 解决：添加负向触发词，更明确描述

- **输出不一致或 API 调用失败/需要用户纠正**：
  - 解决：改进指令，增加错误处理

## 第四章：分发与共享

技能为 MCP 提供了更快的价值实现路径。当前（2026 年 1 月），

**个人用户**：
可以通过下载技能文件夹并在设置中上传来使用：
1. 下载技能文件夹
2. 如有需要将其压缩
3. 在 Claude.ai 中通过“设置 > 功能 > 技能”上传
4. 或将其放入 Claude Code 的 skills 目录

**组织管理员**：
可以跨整个工作区部署技能。
- 管理员可在工作区范围内部署技能（2025 年 12 月 18 日发布）
- 自动更新
- 集中管理

我们已将 [Agent Skills](https://agentskills.io/home) 作为一个开放标准发布。与 MCP 相似，我们认为技能应当可在各种工具和平台之间移植——无论使用 Claude 还是其他 AI 平台，同一个技能都应可用。当然，也有些技能专为特定平台的能力设计；作者可以在兼容性字段中注明这一点。我们正与生态系统的成员共同制定该标准，并对早期采用感到兴奋。

**基于 API 的使用场景：** 对于程序化工作流，可通过 `/v1/skills` 端点进行管理。请注意，在 API 中使用技能需要代码执行工具（Code Execution Tool beta）以提供安全运行环境。
关键功能：
- 使用 `/v1/skills` 端点列出和管理技能
- 在 Messages API 请求中通过 `container.skills` 参数添加技能
- 通过 Claude 控制台进行版本控制与管理
- 可与 Claude Agent SDK 结合，用于构建自定义代理

何时通过 API 调用技能 vs 在 Claude.ai 中使用：

| 用例 | 最佳呈现位置 |
|---|---|
| 终端用户直接与技能交互 | Claude.ai / Claude Code |
| 开发阶段的手动测试与迭代 | Claude.ai / Claude Code |
| 个性化、即席的工作流 | Claude.ai / Claude Code |
| 应用程序以编程方式使用技能 | API |
| 生产级大规模部署 | API（自动化管道与代理系统） |

#### **推荐的分发方式**
建议将技能托管在公开的 GitHub 仓库，并在你的 MCP 文档中加以说明。
1. 将技能托管在 GitHub 上
   - 对于开源技能使用公共仓库
   - 提供清晰的 README 说明安装步骤
   - 包含示例用法和截图
2. 在你的 MCP 仓库中记录
   - 在 MCP 文档中链接技能
   - 说明两者结合使用的价值
   - 提供快速入门指南
3. 编写安装指南
   - **安装 [Your Service] 技能**
     1. 下载技能：
        - 克隆仓库：`git clone https://github.com/yourcompany/skills`
        - 或从发布页面下载 ZIP
     2. 在 Claude 中安装：
        - 打开 Claude.ai > 设置 > 技能
        - 点击“上传技能”
        - 选择技能文件夹（或压缩包）
     3. 启用技能：
        - 切换打开 [Your Service] 技能
        - 确保你的 MCP 服务器已连接
     4. 测试：
        - 询问 Claude：“在 [Your Service] 中设置一个新项目”

#### 宣传skill
应关注它带来的业务成果（如 “取代了 30 分钟的手动设置”），而非解释其底层技术（如 “调用 MCP 工具的 YAML 文件”）。

强调 MCP 与技能的故事：“我们的 MCP 服务器让 Claude 能访问你的 Linear 项目。我们的技能教 Claude 你团队的冲刺规划工作流。两者结合，使 AI 驱动的项目管理成为可能。”

## 第五章：模式与故障排除

技能设计往往分为 “问题优先”（用户描述结果，技能编排工具）或 “工具优先”（技能提供如何使用特定工具的最佳实践）。
可以把它想象成去Home Depot。你可能带着一个问题走进去——“我需要修理厨房橱柜”，店员会指出合适的工具。或者你可能先挑好了一把电钻，然后问如何用它完成具体工作。技能的工作方式与此类似：
- **问题优先**：“我需要设置一个项目工作区”→ 你的技能按正确顺序协调 MCP 调用。用户描述结果，技能处理工具。
- **工具优先**：“我已经连接了 Notion MCP”→ 你的技能教 Claude 最佳工作流和实践。用户已有工具访问，技能提供专业知识。
大多数技能偏向其中一种。了解哪种方式适合你的用例有助于你选择下面的模式。

### 常见架构模式：

#### 模式 1：顺序工作流编排

用于需要严格顺序的多步流程。

```markdown
## Workflow: Onboard New Customer
### Step 1: Create Account
Call MCP tool: `create_customer`
Parameters: name, email, company

### Step 2: Setup Payment
Call MCP tool: `setup_payment_method`
Wait for: payment method verification

### Step 3: Create Subscription
Call MCP tool: `create_subscription`
Parameters: plan_id, customer_id (from Step 1)

### Step 4: Send Welcome Email
Call MCP tool: `send_email`
Template: welcome_email_template
```

Key techniques: • Explicit step ordering • Dependencies between steps • Validation at each stage • Rollback instructions for failures

#### 模式 2：多 MCP 协调  
跨多个服务的工作流。例如，将设计导出（Figma）、资产存储（Drive）、任务创建（Linear）和通知（Slack）分成独立阶段，并传递数据。

示例：设计到开发移交
- 阶段 1：设计导出（Figma MCP）
  1. 从 Figma 导出设计资产
  2. 生成设计规范
  3. 创建资产清单
- 阶段 2：资产存储（Drive MCP）
  1. 在 Drive 中创建项目文件夹
  2. 上传所有资产
  3. 生成共享链接
- 阶段 3：任务创建（Linear MCP）
  1. 创建开发任务
  2. 将资产链接附加到任务
  3. 分配给工程团队
- 阶段 4：通知（Slack MCP）
  1. 在 #engineering 发布移交摘要
  2. 包含资产链接和任务引用

关键技术：
- 清晰的阶段划分
- MCP 之间的数据传递
- 在转移到下一个阶段前进行验证
- 集中错误处理

#### 模式 3：迭代优化  
通过验证脚本自我检查并重试，直到达到质量阈值。

示例：报告生成
- 初稿
  1. 通过 MCP 获取数据
  2. 生成初始报告
  3. 保存到临时文件
- 质量检查
  1. 运行验证脚本：`scripts/check_report.py`
  2. 识别问题：
     - 缺少部分
     - 格式不一致
     - 数据验证错误
- 优化循环
  1. 处理每个问题
  2. 重新生成受影响段落
  3. 重新验证
  4. 重复，直到达到质量阈值
- 最终定稿
  1. 应用最终格式
  2. 生成摘要
  3. 保存最终版本

关键技术：
- 明确的质量标准
- 迭代改进
- 验证脚本
- 知道何时停止迭代

#### 模式 4：上下文感知的工具选择  
当结果相同但根据上下文选择不同工具时使用。

示例：智能文件存储
- 决策树
  1. 检查文件类型和大小
  2. 确定最佳存储位置：
     - 大文件（>10MB）：使用云存储 MCP
     - 协作文档：使用 Notion/Docs MCP
     - 代码文件：使用 GitHub MCP
     - 临时文件：使用本地存储
- 根据决策执行存储：
  - 调用相应的 MCP 工具
  - 应用特定服务的元数据
  - 生成访问链接
- 向用户说明为何选用了该存储

关键技术：
- 明确的决策标准
- 备用方案
- 透明度

#### 模式 5：领域特定智能  
当你的技能在行动之前加入专门知识时使用。

示例：金融合规
- 处理前（合规检查）
  1. 通过 MCP 获取交易详情
  2. 应用合规规则：
     - 检查制裁名单
     - 验证地区许可
     - 评估风险等级
  3. 记录合规决策
- 处理
  - 如果通过合规：
    - 调用支付处理 MCP 工具
    - 应用适当的防欺诈检查
    - 处理交易
  - 否则：
    - 标记待审查
    - 创建合规案例
- 审计追踪
  - 记录所有合规检查
  - 记录处理决策
  - 生成审计报告

关键技术：
- 将领域专业知识嵌入逻辑
- 行动前进行合规
- 全面记录
- 清晰的治理

### 故障排除指南

#### **技能无法上传**
通常是因为文件名不是精确的 `SKILL.md`，或者 YAML 格式有误。

```yaml
# 错误的命名（有空格和大小写）
name: My Cool Skill
# 正确的命名
name: my-cool-skill
```

#### **触发频率异常**
- 如果触发不足，补充更具体的关键词。
- 如果触发过多，在描述中添加负向触发词（如 “Do NOT use for...”）。

#### **MCP 调用失败**
排查步骤：
1. **验证 MCP 服务器连接**
   - Claude.ai：设置 > 扩展 > [你的服务]
   - 应显示“已连接”状态
2. **检查认证**
   - API key 有效且未过期
   - 拥有正确的权限/作用域
   - OAuth 令牌已刷新
3. **独立测试 MCP**
   - 让 Claude 直接调用 MCP（不使用技能）
   - 示例提示：“使用 [服务] MCP 获取我的项目”
   - 若此调用失败，则问题出在 MCP 而非技能
4. **验证工具名称**
   - 确认技能中引用的 MCP 工具名称正确
   - 查阅 MCP 服务器文档确认拼写（名称区分大小写）

#### **上下文过载**
当响应变慢时：
- 将详尽文档移至 `references/` 并使用链接引用
- 缩短 `SKILL.md`（保持在 5000 词以内）
- 减少同时启用的技能数量
   - 工具名称区分大小写

- **上下文过载：** 当响应变慢时，将详细文档移至 `references/` 并缩短 `SKILL.md`（保持在 5000 词以下），并减少同时启用的技能数量。
